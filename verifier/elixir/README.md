# Elixir reference verifier

The Elixir reference verifier for the HumanAccepted receipt format v1.0.0.
Byte-exact with the Python, Go, TypeScript, and Rust reference verifiers.

Uses Erlang's built-in `:crypto.verify/5` with `:eddsa` + `:ed25519` — **no
external dependencies**. The Erlang stdlib is the cryptographic backend.

## Install

```bash
# Erlang/OTP 27+ and Elixir 1.14+ are required.
# macOS:    brew install erlang elixir
# Ubuntu:   apt install elixir erlang
# asdf:     asdf install erlang 27 elixir 1.18
# Docker:   docker run -it --rm -v $(pwd):/app -w /app elixir:1.18 erl
```

## Run the conformance test

```bash
cd verifier/elixir
mix deps.get     # no external deps, but mix needs to set up the project
mix test        # runs 4/4 conformance vectors
```

Or use the CLI binary for a more verbose pass/fail summary:

```bash
mix run_vectors
```

Expected output:

```
Running conformance vectors from vectors/v1
Keys file: vectors/v1/keys.json (4 entries)

  ✓ 01_minimal.json: valid
  ✓ 02_unicode.json: valid
  ✓ 03_all-optional_fields.json: valid
  ✓ 04_unsigned.json: valid

  4/4 vectors pass, 0 failed.
```

If your verifier disagrees with the Python reference on any vector, you
have a bug. Open an issue with the failing vector and the verdict
difference.

## Usage

```elixir
{:ok, pub} = HumanAccepted.Verifier.public_key_from_hex("ed25519:abcdef...")
{:ok, receipt} = Jason.decode(json_string)
verdict = HumanAccepted.Verifier.verify(receipt, pub)
case verdict do
  %{valid: true, reason: "valid"} -> :ok
  %{valid: false, reason: reason} -> {:invalid, reason}
end
```

The verifier is **pure-functional**. No HTTP, no logging, no global
state. `verify/2` is deterministic and side-effect-free.

## What this verifier guarantees

- **Byte-exact canonical form** with the Python, Go, TypeScript, and Rust
  reference verifiers. The same receipt produces the same canonical
  bytes in all 5 implementations.
- **Ed25519 signature verification** via Erlang's `:crypto.verify/5` with
  the `:eddsa` + `:ed25519` algorithm selectors. Pure-function, no key
  material is exposed.
- **Field-shape conformance** (version, signature prefix, `cf_attestation`
  is `null` in v1).
- **The exact same verdict format** as the Python reference:
  `%{valid: boolean, reason: string}`. Cross-language conformance
  assertions can compare verdicts directly.

## What this verifier does NOT do

- **Key generation.** The receipt signer (not the verifier) generates
  keys. Use `:crypto.generate_key(:eddsa, :ed25519)` for testing; a
  real key in an HSM or KMS for production.
- **Canonical-form introspection.** The verifier doesn't expose the
  canonical bytes it computed. If you need to inspect them (for
  example for cross-implementation conformance), call
  `HumanAccepted.Verifier.canonicalize_map/1` directly.
- **Receipt format versioning beyond v1.** pre-v1.0.0 receipts (the
  pre-release) and any future v2+ are rejected at the version check.
  Add support for new versions in a new file, not by forking this one.

## Why Erlang/OTP's `:crypto` and not a NIF?

Erlang/OTP ships Ed25519 verify in `:crypto` since OTP 22. Using it
means:
- **No external dependencies** (no `libsecp256k1`, no `ring`, no Rust
  toolchain). Just `:crypto.verify/5`.
- **FIPS-validated** on supported platforms (Erlang's crypto uses the
  system OpenSSL or AWS-LC backend).
- **Cross-platform** — same code on Linux, macOS, Windows, BSD.

The cost is a small performance hit vs a pure-Rust NIF: ~50µs per
verify on commodity hardware, vs ~5µs for ed25519-dalek. For the
conformance test runner (4 vectors × 1 verify), the difference is
invisible. For a high-throughput hot path, the Rust verifier or a
custom NIF is the right tool.

## Why byte-exact?

Because the Ed25519 signature signs the canonical bytes. If two
implementations produce different canonical bytes for the same logical
receipt, signatures generated by one will fail to verify in the other.
The conformance vectors lock the canonical form down: any
implementation that passes all 4 vectors is byte-exact with the Python
reference.

The Elixir implementation's only real "cleverness" is in
`canonicalize_map/1` — the `Enum.sort_by` uses Elixir's default term
ordering, which (for binaries) is bytewise lexicographic. For ASCII
keys (the common case) this matches Python's `sorted()`. For non-ASCII
keys, Python 3 uses Unicode codepoint sort and Elixir uses bytewise
sort, so a non-ASCII key in a receipt would behave differently. The
spec disallows non-ASCII keys in v1.0.0; if a future version lifts that,
both implementations need to update.

The 4 conformance vectors in `vectors/v1/` are ASCII-only, so this
implementation agrees with the Python reference on every test case.
