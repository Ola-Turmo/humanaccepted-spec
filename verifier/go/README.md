# Go reference verifier

The Go reference verifier for the HumanAccepted receipt format v1.0.0.
Byte-exact with the Python reference on the canonical form, uses the
Go standard library for Ed25519 (no third-party dependencies). Pure
function over (receipt, public_key) → verdict.

## Install

No external dependencies. Just Go 1.22+.

## Usage

```go
import verifier "github.com/Ola-Turmo/humanaccepted-spec/verifier/go"

func main() {
    receiptJSON := []byte(`{...the receipt object...}`)
    var receipt map[string]interface{}
    if err := json.Unmarshal(receiptJSON, &receipt); err != nil { ... }
    pub, err := verifier.PublicKeyFromHexOrBytes("ed25519:<hex>")
    if err != nil { ... }
    v := verifier.Verify(receipt, pub)
    if v.Valid {
        fmt.Println("Receipt is authentic.")
    } else {
        fmt.Println("Invalid:", v.Reason)
    }
}
```

The verifier is **pure-functional**. No HTTP, no logging, no global
state. Map-in, Verdict-out.

## Conformance

The Go verifier is byte-exact-equal in verdict to the Python reference
on every vector in `vectors/v1/`. Run:

```bash
go run test_vectors.go
```

Expected output:

```
Running conformance vectors from <repo>/vectors/v1
Keys file: <repo>/vectors/v1/keys.json (4 entries)

  ✓ 01_minimal.json: valid
  ✓ 02_unicode.json: valid
  ✓ 03_all-optional_fields.json: valid
  ✓ 04_unsigned.json: valid

  4/4 vectors pass, 0 failed.
```

If your Go verifier disagrees with the Python reference on any vector,
you have a bug. Open an issue with the failing vector and the verdict
difference.

## What this verifier guarantees

- **Byte-exact canonical form** with the Python reference. The same
  receipt produces the same canonical bytes in both implementations.
- **Ed25519 signature verification** via `crypto/ed25519` (Go's standard
  library). Pure-function, no key material is exposed.
- **Field-shape conformance** (version, signature prefix, `cf_attestation`
  is `null` in v1).
- **The exact same verdict format** as the Python reference: `{ Valid:
  bool, Reason: string }`. Cross-language conformance assertions can
  compare verdicts directly.

## What this verifier does NOT do

- **Key generation.** The receipt signer (not the verifier) generates
  keys. Use Go's `ed25519.GenerateKey()` for testing; a real key in a
  HSM or KMS for production.
- **Canonical-form introspection.** The verifier doesn't expose the
  canonical bytes it computed. If you need to inspect them (for
  example for cross-implementation conformance), call `canonicalBytes()`
  from `canonical.go` directly.
- **Receipt format versioning beyond v1.** pre-v1.0.0 receipts (the
  pre-release) and any future v2+ are rejected at the version check.
  Add support for new versions in a new file, not by forking this one.

## Why byte-exact?

Because the Ed25519 signature signs the canonical bytes. If two
implementations produce different canonical bytes for the same logical
receipt, signatures generated by one will fail to verify in the other.
The conformance vectors lock the canonical form down: any
implementation that passes all 4 vectors is byte-exact with the Python
reference.

The Go implementation's only real "cleverness" is in `canonical.go`'s
`sortStrings` function — Go's `sort.Strings` uses a locale-aware
ordering for non-ASCII, but Python's `sorted()` uses raw bytewise
ordering. The `sortStrings` function re-implements bytewise sort to
match. The tests confirm: the Go canonical bytes are identical to
the Python canonical bytes for every test vector.
