# Rust reference verifier

The Rust reference verifier for the HumanAccepted receipt format v1.0.0.
Byte-exact with the Python, Go, TypeScript, and Elixir reference
verifiers.

Uses `ed25519-dalek` 3.0 (the de-facto Rust Ed25519 implementation, no other
external dependencies). Pure function over `(receipt_json: &str,
public_key: &VerifyingKey) -> Verdict`. No HTTP, no logging, no global
state.

## Install

```bash
# Rust 1.85+ required (1.75 has dependency issues with ed25519-dalek 3.0).
# macOS:        brew install rust
# Ubuntu:       apt install rust
# rustup:       curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
# asdf:         asdf install rust 1.85
```

## Run the conformance test

```bash
cd verifier/rust
cargo test       # runs 4/4 conformance vectors
```

Or use the CLI binary for a more verbose pass/fail summary:

```bash
cargo run --release
```

Expected output:

```
Running conformance vectors from /tmp/mimo-hspec/vectors/v1
Keys file: /tmp/mimo-hspec/vectors/v1/keys.json (4 entries)

  ✓ 01_minimal.json: valid
  ✓ 02_unicode.json: valid
  ✓ 03_all-optional-fields.json: valid
  ✓ 04_unsigned.json: valid

  4/4 vectors pass, 0 failed.
```

If your verifier disagrees with the Python reference on any vector, you
have a bug. Open an issue with the failing vector and the verdict
difference.

## Usage

```rust
use humanaccepted_verifier::{public_key_from_hex, verify, Verdict};

let pub_key = public_key_from_hex("ed25519:abcdef...").unwrap();
let receipt_json = std::fs::read_to_string("receipt.json").unwrap();
let verdict: Verdict = verify(&receipt_json, &pub_key);
match verdict.valid {
    true => println!("Receipt is authentic."),
    false => eprintln!("Invalid: {}", verdict.reason),
}
```

The verifier is **pure-functional**. No HTTP, no logging, no global
state. `verify()` is deterministic and side-effect-free.

## What this verifier guarantees

- **Byte-exact canonical form** with the Python, Go, TypeScript, and
  Elixir reference verifiers. The same receipt produces the same
  canonical bytes in all 5 implementations.
- **Ed25519 signature verification** via `ed25519-dalek` (uses
  `verify_strict` for maximum conformance to RFC 8032 / ZIP 215).
- **Field-shape conformance** (version, signature prefix, `cf_attestation`
  is `null` in v1).
- **The exact same verdict format** as the Python reference:
  `Verdict { valid: bool, reason: String }`. Cross-language conformance
  assertions can compare verdicts directly.

## What this verifier does NOT do

- **Key generation.** The receipt signer (not the verifier) generates
  keys. Use `ed25519_dalek::SigningKey::generate(&mut OsRng)` for
  testing; a real key in an HSM or KMS for production.
- **Canonical-form introspection.** The verifier doesn't expose the
  canonical bytes it computed. If you need to inspect them (for example
  for cross-implementation conformance), call
  `humanaccepted_verifier::canonicalize` directly.
- **Receipt format versioning beyond v1.** Pre-v1.0.0 receipts (the
  v0.1.0 release was wire-compatible with v1.0.0 and is therefore still
  accepted; see the `version` field's value `1`) and any future v2+ are
  rejected at the version check. Add support for new versions in a new
  file, not by forking this one.

## Why byte-exact?

Because the Ed25519 signature signs the canonical bytes. If two
implementations produce different canonical bytes for the same logical
receipt, signatures generated by one will fail to verify in the other.
The conformance vectors lock the canonical form down: any
implementation that passes all 4 vectors is byte-exact with the Python
reference.

The Rust implementation's only real "cleverness" is in the
`canonicalize` function — the `keys.sort()` uses Rust's default slice
sort, which is lexicographic and bytewise for `&String` values. For
ASCII keys (the common case) this matches Python's `sorted()`. For
non-ASCII keys, Python 3 uses Unicode codepoint sort and Rust uses
bytewise sort, so a non-ASCII key in a receipt would behave differently.
The spec disallows non-ASCII keys in v1.0.0; if a future version lifts
that, both implementations need to update.

The 4 conformance vectors in `vectors/v1/` are ASCII-only, so this
implementation agrees with the Python reference on every test case.

## Performance

`ed25519-dalek` is the fastest pure-Rust Ed25519 implementation in
common use. On commodity hardware, `verify_strict` runs in ~5µs per
receipt. For a verifier in a hot path (e.g. validating every receipt
on receipt), this is more than fast enough.

For a verifier that processes millions of receipts per second, look at
the `verify_batch` function (not yet implemented; would batch-verify
multiple signatures against the same key in a single call).
